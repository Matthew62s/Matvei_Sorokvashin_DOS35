Тема: Исследование системы непрерывной интеграции (Continuous Integration) на основе Jenkins
В современных условиях разработки программного обеспечения особое значение приобретают методы автоматизации процессов сборки, тестирования и развертывания приложений. Одной из наиболее распространённых практик является непрерывная интеграция (Continuous Integration, CI), которая позволяет своевременно выявлять ошибки, повышать качество программного обеспечения и ускорять процесс разработки.
Системы непрерывной интеграции автоматически выполняют сборку проекта и запуск тестов после внесения изменений в исходный код. Это позволяет разработчикам быстро получать информацию о возникших ошибках и своевременно их устранять.
Одной из наиболее популярных систем непрерывной интеграции является Jenkins. Благодаря открытому исходному коду, большому количеству плагинов и возможности интеграции с различными инструментами разработки Jenkins широко используется как в небольших проектах, так и в крупных организациях.

Цель работы — изучить принципы непрерывной интеграции и рассмотреть возможности системы Jenkins.
Для достижения поставленной цели необходимо решить следующие задачи:

Изучить понятие Continuous Integration;
Рассмотреть основные принципы CI;
Сравнить наиболее популярные системы непрерывной интеграции;
Ознакомиться с возможностями Jenkins;
Выполнить практическую настройку системы Jenkins.

1 Теоретические основы Continuous Integration
1.1 Понятие Continuous Integration

Continuous Integration (CI) — это практика разработки программного обеспечения, при которой изменения в исходном коде регулярно объединяются в общий репозиторий. После каждого изменения автоматически выполняются сборка проекта и запуск тестов, что позволяет быстро обнаружить ошибки и предотвратить их накопление.
До появления CI разработчики зачастую объединяли изменения только после завершения крупных этапов работы. Такой подход нередко приводил к конфликтам в коде, длительной интеграции и сложному поиску ошибок. Использование непрерывной интеграции позволяет значительно сократить подобные проблемы за счёт регулярного объединения изменений и автоматической проверки работоспособности проекта.
Основной задачей CI является обеспечение стабильного состояния проекта на всех этапах разработки. При обнаружении ошибки система уведомляет разработчиков, что позволяет оперативно устранить проблему и предотвратить её распространение.
Сегодня технологии непрерывной интеграции применяются практически во всех современных проектах, начиная от небольших веб-приложений и заканчивая крупными корпоративными информационными системами.
1.2 Основные принципы непрерывной интеграции
Эффективность использования CI достигается благодаря соблюдению нескольких основных принципов.
Первым принципом является регулярная интеграция изменений. Разработчики как можно чаще отправляют изменения в общий репозиторий, что уменьшает вероятность возникновения конфликтов при объединении кода.
Второй принцип — автоматическая сборка проекта. После каждого изменения система самостоятельно запускает процесс сборки приложения без участия разработчика.
Следующим важным принципом является автоматическое тестирование. После успешной сборки запускаются тесты, позволяющие проверить корректность работы программы и убедиться, что внесённые изменения не нарушили существующую функциональность.
Ещё одним принципом является оперативная обратная связь. В случае возникновения ошибок разработчики сразу получают уведомление о неудачной сборке или тестировании, что позволяет быстро приступить к исправлению проблемы.
Кроме того, важным элементом CI является использование системы контроля версий, например Git, которая обеспечивает совместную работу нескольких разработчиков и хранение истории изменений проекта.
Соблюдение перечисленных принципов позволяет существенно повысить качество программного обеспечения, сократить время разработки и уменьшить количество ошибок, возникающих при интеграции изменений.
1.3 Преимущества и недостатки Continuous Integration
Использование непрерывной интеграции обладает рядом преимуществ.
К основным достоинствам относятся:
быстрое обнаружение ошибок;
повышение качества программного обеспечения;
сокращение времени разработки;
автоматизация процессов сборки и тестирования;
уменьшение количества конфликтов при объединении изменений;
возможность постоянного контроля состояния проекта.
Несмотря на многочисленные преимущества, технология CI имеет и определённые недостатки. Для её внедрения требуется первоначальная настройка инфраструктуры, установка необходимых инструментов и поддержка сервера непрерывной интеграции. Кроме того, автоматические тесты должны регулярно обновляться по мере развития проекта.
В целом преимущества непрерывной интеграции значительно превышают её недостатки, поэтому сегодня CI является одним из стандартов современной разработки программного обеспечения.
2 Сравнение современных CI-систем
В настоящее время существует большое количество систем непрерывной интеграции. Наиболее распространёнными являются Jenkins, GitLab CI/CD, CircleCI и Travis CI. Каждая из них обладает своими особенностями и предназначена для автоматизации процессов сборки, тестирования и развертывания приложений.
Jenkins является одной из самых популярных CI-систем благодаря открытому исходному коду и большому количеству доступных плагинов. Он поддерживает интеграцию практически с любыми системами контроля версий, средствами тестирования и инструментами развертывания. Jenkins может работать как на физических серверах, так и в виртуальной среде или контейнерах Docker.
GitLab CI/CD встроен непосредственно в платформу GitLab и не требует установки отдельного сервера для организации базового процесса непрерывной интеграции. Настройка выполняется с помощью файла .gitlab-ci.yml, что делает работу с системой достаточно удобной.
CircleCI представляет собой облачную платформу непрерывной интеграции. Она отличается простотой настройки, высокой скоростью выполнения сборок и поддержкой большого количества современных языков программирования. Однако бесплатная версия имеет определённые ограничения по использованию ресурсов.
Travis CI также является облачным сервисом непрерывной интеграции. Он долгое время широко использовался для проектов с открытым исходным кодом благодаря простой настройке и интеграции с GitHub. В настоящее время его популярность снизилась из-за изменений условий использования и появления более функциональных альтернатив.

## Установим jenkins

Создадим такой Dockerfile

FROM jenkins/jenkins:2.568.1-jdk21
USER root
RUN apt-get update && apt-get install -y lsb-release ca-certificates curl && \
    install -m 0755 -d /etc/apt/keyrings && \
    curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc && \
    chmod a+r /etc/apt/keyrings/docker.asc && \
    echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
    https://download.docker.com/linux/debian $(. /etc/os-release && echo \"$VERSION_CODENAME\") stable" \
    | tee /etc/apt/sources.list.d/docker.list > /dev/null && \
    apt-get update && apt-get install -y docker-ce-cli && \
    apt-get clean && rm -rf /var/lib/apt/lists/*
USER jenkins
RUN jenkins-plugin-cli --plugins "blueocean docker-workflow json-path-api"

## Сбилдим и запустим его.

docker run \
  --name jenkins-blueocean \
  --restart=on-failure \
  --detach \
  --network jenkins \
  --env DOCKER_HOST=tcp://docker:2376 \
  --env DOCKER_CERT_PATH=/certs/client \
  --env DOCKER_TLS_VERIFY=1 \
  --publish 8080:8080 \
  --publish 50000:50000 \
  --volume jenkins-data:/var/jenkins_home \
  --volume jenkins-docker-certs:/certs/client:ro \
  myjenkins-blueocean:2.568.1-1

## Создадим simple pipeline
```groovy
pipeline {
    agent any
    stages {
        stage('Deploy') {
            steps {
                sshagent(credentials: ['ssh_private']) {
                    sh """
                        ssh -T -o StrictHostKeyChecking=no -p 8392 root@62.181.53.80 '
                        set -e
                        cd /opt/jenkins_workdir
                        if [ ! -d Matvei_Sorokvashin_DOS35 ]; then
                            git clone --filter=blob:none --no-checkout \
                                git@github.com:Matthew62s/Matvei_Sorokvashin_DOS35.git
                        fi
                        cd Matvei_Sorokvashin_DOS35
                        git sparse-checkout init --cone
                        git sparse-checkout set Lesson24_Docker_HW24/Dynamic_service
                        git checkout
                        git pull
                        cd Lesson24_Docker_HW24/Dynamic_service
                        docker compose up --build -d
                        '
                    """
                }
            }
        }
    }
}
```
